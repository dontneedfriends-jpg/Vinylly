# AGENTS.md — Vinylly

Руководство для агентов, работающих над приложением **Vinylly** — каталогизатором физической аудио-коллекции (винил, CD, кассеты, другие носители).

## Описание проекта

Vinylly — приложение для учёта физической коллекции аудио-носителей. Пользователь добавляет релизы в коллекцию, приложение автоматически подгружает обложки, треклисты и (по возможности) тексты песен. Интерфейс должен быть красивым и удобным: акцент на визуальную часть (обложки, плитки релизов), быстрый поиск, фильтрацию и группировку.

## Целевые платформы

- **Основные (MVP):** нативные десктоп-сборки под **Linux** и **Windows**, распространяемые как portable-артефакт (один исполняемый файл / папка без установки в систему).
- **Перспективные (лёгкий порт):** веб (та же кодовая база, сборка в PWA/SPA), **Android** и **iOS**. Архитектура должна допускать эти порты без переписывания доменной логики и UI — менять только слой хоста (фс, сеть, окна).
- Портируемость — приоритет №1: бизнес-логика, данные и UI не должны зависеть от конкретной ОС или рантайма.

### Стратегия портируемости

- **Единый UI на React** — рендерится внутри нативного окна на десктопе и в WebView на мобильных; в вебеstorybook/Vite-сборка. Никаких платформенно-специфичных виджетов в UI-слое.
- **Нативная оболочка — Tauri 2.x**: кросс-компиляция Linux/Windows из одной кодовой базы, малый размер бинаря, перенос путей к данным per-OS (см. ниже). Tauri 2 поддерживает Android/iOS — путь к мобильному порту без смены стека.
- **Тонкий хост-слой**: всё, что касается файловой системы, путей, окон, сторонних библиотек, изолировано за интерфейсами (`HostFs`, `HostPaths`, `HostNet`), реализуемыми per-платформой. UI и домен работают только с этими интерфейсами.
- **Portable-режим данных**: при запуске с флагом `--portable` (или `VINYL_PORTABLE=1`) все данные (БД, кеш обложек, кеш API) пишутся в папку рядом с исполняемым файлом (`./data`), а не в системный appdata. Без флага — стандартные директории ОС (`%APPDATA%`, `~/.local/share`, `~/Library/Application Support`).
- **Офлайн-first**: приложение полностью функционально без сети; внешние API (Discogs/MB/Genius) — только обогащение, не источник истины. Запросы кэшируются, повторные обращения берутся из локального кеша.
- **Бинарник без зависимостей**: целевой артефакт — один файл (Tauri собирает Rust + WebView payload), не требующий установки рантайма на машине пользователя. На Linux/Windows — WebView2 (Win) / WebKitGTK (Linux) берётся из системы.
- **Никаких системных инсталляций в коде**: никаких `.deb`/`.msi`-специфичных шагов в сборке — только portable-директория; установщики опциональны сторонними средствами (msi-wrapveler, AppImage) на этапе полировки.

## Технологический стек

- **Frontend**: React + TypeScript, Vite.
- **UI / дизайн-система**: Tailwind CSS + **дизайн-система Neumorphism**, описанная в `./neumorphism/` (см. раздел ниже). Поддержка тёмной/светлой темы автоматическая через `prefers-color-scheme`.
- **State/данные**: TanStack Query (серверный кеш, офлайн-очередь) + Zustand (локальный UI-state).
- **Бэкенд/логика**: бизнес-логика в виде чистых TypeScript-модулей (язык- и рантайм-нейтральные), вызываемых из UI и из Rust-стороны Tauri. Отдельный Node-бэкенд НЕ нужен — всё работает in-process внутри Tauri. (Веб-порт при необходимости поднимает тот же TS-код в браузере через WASM/SQLite-wasm или тонкий worker.)
- **ORM/данные**: Prisma ORM с драйвером better-sqlite3 на десктопе (node-side Tauri), в веб-порту — `@prisma/client` поверх sqlite-wasm. Схема одна.
- **Хранилище**: SQLite, файл БД лежит в portable-директории данных. Никаких внешних серверов БД. Миграция на PostgreSQL — не приоритет,uffixовательно для single-user.
- **Нативная оболочка**: Tauri 2.x (Rust-core, WebView-frontend). Команды и события — через `tauri::command`,.setMaximum размера/icallável.
- **Поиск медиа/обложек**: Discogs API (основной), MusicBrainz + Cover Art Archive (fallback), Last.fm (доп. метаданные). Доступ только через `packages/media-providers`; фронтенд/WebView не зовёт провайдеров напрямую, все HTTP-запросы идут через Rust-сторону (чтобы корректно работать с прокси/сертами системы и обходить CORS в веб-порту).
- **Тексты песен**: Genius API (требует токен), публичные тексты через LRCLIB/прочее — только при наличии легального источника; иначе поле ручного ввода.
- **Кэш обложек**: локальная файловая папка (`data/covers`) + миниатюры; мгновенный показ из кеша. Хранение путём в HostFs, не в БД.
- **Тесты**: Vitest (unit), Playwright/WebDriverIO (e2e по WebView). Rust-сторона — `cargo test`.
- **Линт/формат**: ESLint + Prettier; `npm run lint` и `npm run typecheck` обязательны перед коммитом. Rust: `cargo fmt` + `cargo clippy`.

## Визуальный язык — Neumorphism

Весь UI **строго** следует дизайн-системе Neumorphism, описанной в `./neumorphism/`. Это не опция и не «как-нибудь» — это контракт на внешний вид приложения. Цель: интерфейс, который выглядит сделанным дизайнером, а не «вайбкодингом» (то есть не дефолтный HTML, не.Material/MUI внешний вид, не голый shadcn).

### Обязательные правила

1. **Перед написанием любого UI-кода** — прочитать релевантные модули из `./neumorphism/`. Минимум для любой задачи: `SKILL.md`, `brand.md`, `colors.md`, `typography.md`, `layout.md`, `radius.md`, `shadows.md`, `borders.md`. Для конкретных компонентов — соответствующий модуль (`buttons.md`, `cards.md`, `inputs.md`, `modals.md` и т.д.). Список в `neumorphism/SKILL.md`.
2. **Токены — не Tailwind-классы.** Имена токенов в `.md` (`neutral-primary-soft`, `heading`, `shadow-md`, `border-default`) — это агностичные дизайн-токены. Их необходимо явно смаппить на CSS custom properties / Tailwind theme config в `apps/web/src/styles/` перед использованием. Не использовать `bg-neutral-primary-soft` «как есть», если не определён в конфиге.
3. **Neumorphic depth — из теней, а не из фона.** Конструктивные элементы (карточки, кнопки, инпуты) **делят цвет фона с родительским контейнером** (низкий контраст) и получают объём за счёт двусторонних теней (`shadow-sm…shadow-2xl` — светлая тень сверху-слева, тёмная снизу-справа). Запрещено лепить высококонтрастные сплошные фоны для структурных элементов.
4. **Палитра и тени фиксированы.** Использовать только токены из `colors.md`, `shadows.md` (с учётом тёмного режима: тёмные/светлые варианты цвета поверхности `#262833` вместо `#b8b9be/#ffffff`). Никаких кастомных `box-shadow` значений и произвольных hex вне палитры.
5. **Тёмный режим — автоматический** через `@media (prefers-color-scheme: dark)` поверх CSS custom properties. Никогда вручную не менять цвета в коде в зависимости от темы.
6. **Состояния интерактива — обязательно.** Каждый интерактивный элемент имеет `:hover`, `:focus`, `:disabled` согласно своему модулю. Hover/focus поднимает elevation на один шаг (например `shadow-sm → shadow-md`), pressed → `shadow-inset`.
7. **HTML — семантический.** Корректная иерархия заголовков `<h1>→<h6>`, `<button>` для действий, `<a>` для навигации, ARIA где нужно. Это часть не «вайбкод-вида».
8. **Радиусы вложенные.** Радиус внутреннего элемента меньше радиуса контейнера на величину внутреннего padding (`radius.md`) — даёт тот самый выточенный neumorphic вид, а не «карточка в карточке».

### Что считается «вайбкодингом» и недопустимо

- Дефолтные Tailwind-цвета (типа `bg-slate-800`, `text-blue-500`) поверх neumorphic поверхности без маппинга на токены.
- Карточки со сплошной заливкой и одной направленной тенью (как Material).
- Прямоугольные `rounded-md` повсюду без учёта вложенности.
- Кнопки, меняющие цвет фона на on-brand при hover вместо смены elevation.
- Использование Tailwind `shadow-*` (его дефолтных теней) вместо neumorphic двусторонних токенов.
- Готовые компоненты shadcn/ui **без перекрашивания** под neumorphic систему — если shadcn используется, его primitives проходят через обёртку, применяющую neumorphic тени/радиусы/отступы.

## Доменная модель (минимальная)

- **Collection** — коллекция пользователя (по умолчанию одна).
- **Item** — единица коллекции.
  - `id`, `type` (vinyl | cd | cassette | other), `barcode`, `catalogNumber`
  - `condition` (sleeve + media), `notes`, `acquiredAt`, `location`, `tags[]`
  - связь с `Release` (один ко многим: один релиз может быть в нескольких копиях/форматах)
- **Release** — каталогизированный релиз (из Discogs/MB).
  - `sourceId`, `source` (discogs|musicbrainz|manual), `title`, `artist`, `year`, `genre[]`, `style[]`, `coverUrl`, `thumbUrl`
  - `tracklist[]` → Track
- **Track** — `position`, `title`, `duration`, `lyrics` (nullable), extra fields (lyrics проверяются/сопоставляются отложенно).
- **Artist** — нормализованный артист (id из MB/Discogs; витрина только при нужде).

## Ключевые сценарии (MVP)

1. **Добавить релиз** — пользователь вводит штрих-код / каталожный номер / название → поиск по провайдерам → выбор совпадения → импорт релиза в локальную БД (маппинг полей, скачивание и кеширование обложки).
2. **Просмотр коллекции** — сетка плиток с обложками; фильтры по формату/году/жанру/артисту; сортировка; поиск по тексту.
3. **Детали релиза** — обложка (зум), треклист с длительностью, текст песни по выбранному треку, метаданные, мои заметки/состояние/теги.
4. **Редактирование** — правка метаданных, замена обложки (загрузка своей), ручной ввод релиза при отсутствии в источниках.
5. **Экспорт/импорт** — JSON/CSV + бэкап БД.

## План разработки (этапы)

### Этап 0 — Скаффолд (1)

- Монорепо (pnpm workspaces), базовые конфиги (TS, ESLint, Prettier, Vitest).
- PR-шаблон, скрипты `lint`, `typecheck`, `test`, `dev`, `build`.

### Этап 1 — База данных и модели (2)

- Prisma-схема под доменную модель.
- Миграции, сидер с тест-данными.
- Репозитории (CRUD) с тестами.

### Этап 2 — Провайдеры медиаданных (3)

- Клиент Discogs (поиск релиза, get release, get master).
- Клиент MusicBrainz + Cover Art Archive.
- Клиент Last.fm (доп. метаданные/биографии).
- Клиент Genius (поиск текстов) — **опционально**, включается флагом/токеном.
- Согласованный интерфейс `MediaProvider` (search/byBarcode/byCatalogId/getRelease/getCover/getLyrics).
- Локальный кеш ответов (TTL) — избегать повторных запросов.

### Этап 3 — Слой данных и хост-команды (3)

- Tauri-команды (`tauri::command`) для операций с коллекцией, items, releases, tracks.
- Загрузка/кеширование обложек через Rust-сторону (хранение файлов в `data/covers` через HostFs).
- Реализации `HostFs`/`HostPaths`/`HostNet` для desktop (Tauri) и веба (fallback на fetch/IndexedDB).
- Авторизация оставлена на будущее; пока single-user.

### Этап 4 — Фронтенд: коллекция (4)

- Лэйаут + тема + навигация.
- Сетка коллекции (виртуализация на больших списках).
- Детальная страница релиза, плеера нет — только каталогизация.
- Формы добавления/редактирования.
- Поиск и фильтры.

### Этап 5 — Тексты песен (4)

- Интеграция Genius с fallback на ручной ввод.
- Хранение текстов в БД (с указанием источника/лицензии).
- UI отображения текста по треку; синхронизированные тексты (lrc) при наличии — показ как бонус.

### Этап 6 — Полировка (4)

- Анимации, скелетоны, empty states.
- Импорт/экспорт, бэкапы.
- Темизация, иконки носителей, бейджи состояния.
- Тесты e2e на ключевые сценарии.

### Этап 7 — (опционально) Десктоп/мобильная поставка

- Tauri-сборка под Linux/Windows, portable SQLite.
- APK/IPA-сборка при наличии ресурсов; Mauriceуром portable-режима на мобильных — sandboxed app storage.

## Конвенции

- Язык кода/комментариев — английский; пользовательский UI — русский (с возможностью i18n).
- Компоненты — функциональные, хуки; бизнес-логика в сервисах/репозиториях.
- Доступ к внешним API — только через `media-providers`; фронтенд не зовёт провайдеров напрямую.
- Никаких секретов в репозитории: токены — через `.env` (пример `.env.example`).
- Чувствительные данные (токены, личные заметки) не логируются.
- Перед коммитом**: `pnpm lint && pnpm typecheck && pnpm test`.
- Коммиты — Conventional Commits (`feat:`, `fix:`, `chore:` и т.д.).
- Rust-сторона: `cargo fmt` + `cargo clippy` без warningов.

## Чего следует избегать

- Не реализовывать стриминг/воспроизведение аудио — цель каталогизация, не плеер.
- Не тащить тяжёлые ORM-альтернативы; держать Prisma.
- Не хардкодить токены или данные провайдеров.
- Не делать собственный парсинг Discogs/Genius, превышающий их ToS; при лимитах — честная заглушка с ручным вводом.
